# occluder-mesh-driver.ts

## Purpose

Main-thread orchestration for the occluder Web Worker offload — the glue between the occluder wiring and a worker running `runMeshRequest`. THREE-free + framework-owned; a consumer supplies a `Worker` adapter and applies the result (e.g. `OcclusionMesh.applyMeshData`).

## Public API

- `new OccluderMeshDriver(poster: MeshWorkerPoster | null, options?)` — `null` ⇒ **synchronous fallback** (meshes inline, calls back immediately). `options`: `onWorkerUnusable?()` (the driver gave up on the worker → consumer should terminate it), `onError?(err)` (a job aborted — worker error event or a sync-meshing throw — for logging), `onMeshStats?(stats)` (per-completed-job timing/size, see below), and `now?()` (injectable ms clock for the stats, default `performance.now()`).
- `OccluderMeshStats` — `{ durationMs, cellCount, mode, synchronous }` reported via `onMeshStats` when a job completes (fires just before that job's `onMesh`). `durationMs` is wall-clock post→geometry (inline time on the sync path) — i.e. the occluder's current refresh latency, the freshness number the Phase-2 gate's field walk reads (plan §"Next step"). Failed/dropped jobs report **no** stats (failures surface via `onError`; coalesced-away intermediates never ran).
- `request(cells, cellSizeM, mode, getCellPoint, onMesh)` — pack + post a mesh job; `onMesh(positions, indices)` fires when it returns. `cells` is the tuple array or a flat `Int32Array` snapshot (`OccupancyGrid.getOccupiedCellsFlat`, Step 1.3 of the 2026-07-03 fps plan); a flat snapshot must be fresh — packing transfers/detaches its buffer.
- `busy` — a job is in flight (worker path).
- `dispose()` — detach the worker handlers, drop any pending job.
- `MeshWorkerPoster` — the minimal `Worker`-like surface (`postMessage(message, transfer)` + `onmessage` + `onerror`) the driver drives; `OnMesh` — the result callback; `OccluderMeshDriverOptions` — the optional callbacks above.

## Invariants & assumptions

- **Coalesce to latest:** at most ONE job in flight. A `request` made while busy becomes the single `pending` job (newest wins); intermediates are dropped — so work never queues behind a growing grid. On response, the completed job's `onMesh` fires, then the pending job (if any) is posted.
- **Callbacks run with the slot still reserved (PR #147 review, fixed 2026-07-04):** `onMeshStats`/`onMesh` (in `handleResponse`) and `onError` (in `failInFlight`) are invoked BEFORE the in-flight slot clears, so a consumer callback that synchronously calls `request()` coalesces into `pending` under the same newest-wins policy — previously it saw an idle driver, posted immediately, and the subsequent pending drain double-posted, dropping the reentrant job's response as stale. The clear + pending drain sit in a `finally`, so a **throwing** callback can neither wedge the slot nor strand the pending job (it still posts).
- **Sync fallback delivers immediately** (no coalescing needed — each `request` completes before returning).
- **Post-dispose safety:** a response arriving after `dispose()` (or with a stale id) is ignored; no callback fires.
- The driver does not own the worker lifecycle — the consumer terminates the `Worker` (see the recorder's `occluder-mesh-worker-client.ts`).
- **Error recovery (never wedges — both orderings).** A worker error is routed to the driver via `poster.onerror`, and `handleWorkerError` recovers whether or not a job is in flight:
  - **Error _while a job is in flight_ (`inFlightId !== null`** — the ordering of an **uncaught throw in `runMeshRequest`**): the driver clears the in-flight slot and re-posts the coalesced `pending` snapshot if one is queued (else the next `request` posts normally) — it never re-posts the _failed_ job (deterministic bad data would loop). If the worker had never meshed (`hasSucceeded` still false) it is declared unusable (`onWorkerUnusable`) and the driver switches permanently to synchronous meshing; a worker that has meshed once is treated as transient and kept.
  - **Error _before the first `request`_ (`inFlightId === null`** — the realistic ordering for a **module-load failure**, which errors within tens of ms, _before_ the throttled first refresh posts anything): a load-failed worker silently discards later posts (no second error), so ignoring this would wedge the first post forever. If the worker has **never** succeeded (`!hasSucceeded`) and we are not already in `syncMode`, the driver declares it unusable (`onError` + `onWorkerUnusable`) right here and switches to synchronous meshing, so the first `request` never touches the dead worker. A stray/late error on a worker that **has** meshed once is transient — kept, ignored.
  - **Synchronous-throw guards (all three sites in the post path):** the synchronous `runMeshRequest` path guards its throws (e.g. bad `cellSizeM`), **and** the worker `poster.postMessage` call is guarded the same way — `Worker.postMessage` can throw synchronously (`DataCloneError` for a non-cloneable payload, or an already-detached/invalid transferable), and since the slot is marked in-flight _before_ the post, an unguarded throw would wedge the driver forever. Both route to `failInFlight`, so they report via `onError` and clear the slot; a post that throws before the worker ever meshed is treated as an unusable worker (→ sync fallback), matching the load-failure semantics above. **`packMeshRequest` is guarded too** (PR #152 review): packing runs the caller's `getCellPoint` per cell (surface modes) and throws on an invalid flat snapshot, after the slot is already marked in-flight. It routes to `failInFlight` with `workerAtFault: false` — a pack-time throw happens on the main thread _before_ the worker is involved, so it reports via `onError` and clears the slot **without** condemning a healthy worker to the sync fallback (unlike the postMessage case).

## Tests

- `occluder-mesh-driver.test.ts` — sync fallback matches a direct mesh; posts + delivers on response; coalesces to the latest (drops intermediates); no delivery after dispose; **error recovery** — a transient error re-posts the pending snapshot, an error with nothing queued un-wedges so the next request posts (the freeze regression guard), a first-error-before-success (job in flight) falls back to sync (`onWorkerUnusable` fired), a **pre-first-post error** (`inFlightId === null`, never succeeded) also falls back to sync without ever posting to the dead worker (the module-load-race regression guard), a **stray error after a prior success** with nothing in flight is ignored (proven-good worker kept), a sync-meshing throw reports via `onError` and recovers, a sync-path CONSUMER onMesh throw is reported via `onError` instead of silently swallowed (the mesh succeeded; the slot check distinguishes consumer-callback errors from mesh failures), and a **throwing `getCellPoint` during packing** reports via `onError`, clears the slot (not wedged) and keeps the worker — both after a prior success and on the very first request (`onWorkerUnusable` never fired for a main-thread provider error); **reentrancy** — a synchronously re-requesting `onMesh` or `onError` coalesces (one follow-up post, newest wins, the reentrant job's geometry delivered), and a throwing `onMesh` still clears the slot and drains the pending job. Uses a fake poster (no worker env needed). **Mesh stats** — a completed worker job reports `{durationMs, cellCount, mode, synchronous:false}` via `onMeshStats` (fake clock), the sync fallback reports `synchronous:true`, coalescing reports one stats entry per _completed_ job (dropped intermediates report nothing), and a failed job reports nothing.
